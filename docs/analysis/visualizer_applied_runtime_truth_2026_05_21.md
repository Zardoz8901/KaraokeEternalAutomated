# Visualizer Applied Runtime Truth — 2026-05-21

## Context

Phase 7A establishes a truthful Player-applied visualizer state before the separate Phase 7B video-source sync work. Before this slice, Orchestrator could infer sent/synced code, selected preset, and local preview load, but it could not know which visualizer the Player actually evaluated and rendered.

## Decision

- The server assigns accepted Hydra broadcasts a server-owned `visualizerRunId`, `visualizerCodeHash` using SHA-256, and `visualizerAcceptedAt`.
- The Player runtime has a `playerInstanceId` generated by `PlayerController` on mount.
- `PLAYER_EMIT_VISUALIZER_APPLIED` is accepted only from the currently pinned Player identity `{ socketId, playerInstanceId }`.
- Applied events are ignored when malformed, oversized, unknown, stale, superseded, or sent by the wrong Player identity.
- `HydraVisualizer` emits applied once per run id only after eval succeeds and the first tick succeeds.
- Orchestrator maps saved preset and gallery applied metadata back to stable row keys, keeping `Selected`, `Loaded in preview`, `Sent/Synced`, and `Applied on Player` separate.

## REDTEAM

- Client-provided run ids, hashes, and accepted timestamps are ignored. The server owns runtime identity and run metadata.
- A manager socket that is not the pinned Player identity cannot overwrite applied state.
- A stale Player applied event for an older run id cannot overwrite a newer accepted visualizer run.
- Raw Hydra code remains executable by managers, but raw applied events do not map to preset rows.
- Gallery applied matching depends on stable gallery metadata; this slice adds `hydraGalleryId` for Orchestrator gallery sends and Player runtime gallery cycling.
- This does not sandbox Hydra eval or change the existing manager trust boundary.
